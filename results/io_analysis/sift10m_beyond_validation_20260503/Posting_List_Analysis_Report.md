#### 2.2.1 空 Posting (0 page) 分析

**现象**: 索引中存在 page count = 0 的 posting（SIFT1M: 659, SIFT10M: 45）

**根因分析**:

1. **K-means 聚类特性**: SPANN 使用 K-means 对向量进行聚类，每个聚类中心对应一个 posting list。某些聚类中心可能因为位置偏远或数据分布不均，没有吸引到任何向量。

2. **代码逻辑** (ExtraStaticSearcher.h lines 4740-4750):
   ```cpp
   // 如果 posting list 大小为 0，listPageCount 保持为 0
   if (p_postingListSizes[i] == 0) {
       listPageCount = 0;  // 空 posting
   }
   ```

3. **规模对比**:
   | 数据集 | 空 posting 数 | 总 posting 数 | 空 posting 比例 |
   |--------|--------------|---------------|----------------|
   | SIFT1M | 659 | 150,076 | **0.44%** |
   | SIFT10M | 45 | 1,496,408 | **0.003%** |

4. **对查询的影响**: **无影响**
   - 空 posting 永远不会被路由选中（没有向量属于该聚类）
   - 不会产生 I/O 开销
   - 只是索引元数据中的"死空间"

5. **为什么 SIFT10M 空 posting 比例更低？**
   - SIFT10M 数据分布更均匀
   - K-means 在更大规模数据上聚类质量更高
   - 聚类中心数与向量数的比例更优（1:6.7 vs 1:6.7，但绝对数量更多）

| 统计量 | SIFT1M | SIFT10M |

| 平均访问次数 | ~4.3 | 1.28 |
| P50 | ~4 | 1 |
| P99 | ~12 | 3 |
| 最大 | ~150 | 9 |

**关键洞察**:
- **77% 的 posting 只被访问 1 次** (SIFT10M)
- 这意味着 cache 的 cross-query reuse 潜力极低

### 3.3 Cross-Query Reuse 公式

```
期望访问次数 = (查询数 × 每查询访问的 posting 数) / 被访问的 posting 总数
           R = (Q × K) / N
```

| 数据集 | Q | K | N | R (期望访问次数) |
|--------|---|---|---|------------------|
| SIFT1M | 10,000 | 64 | 98,000 | **6.5 次** |
| SIFT10M | 10,000 | 64 | 498,122 | **1.28 次** |

**结论**: SIFT10M 的 cross-query reuse 是 SIFT1M 的 **1/5**

## 7. Cross-Query Reuse 详细分析

### 7.1 定义与测量

**Cross-Query Reuse**: 同一个 posting 被多个不同查询访问的比例

| 类型 | SIFT10M 数量 | 占比 | 总 I/O wait |
|------|--------------|------|-------------|
| 单查询访问 | 383,450 | 77.0% | 289,748 ms |
| 多查询访问 | 114,672 | 23.0% | 195,017 ms |

### 7.2 单页 Posting 的 Cross-Query Reuse

| 指标 | SIFT1M | SIFT10M | 比值 |
|------|--------|---------|------|
| Single-page postings 数量 | ~30,000 | 127,457 | 4x |
| 平均访问次数 | 134.9 | 1.27 | **1/106** |
| 只访问 1 次的比例 | ~5% | 78.2% | 15x |

**这是 M1 Page Cache 失效的根本原因**:
- SIFT1M: 134.9 次平均访问 → 高 cache 复用
- SIFT10M: 1.27 次平均访问 → 几乎无 cache 复用

### 7.3 Cache 效率模型

**前提条件**: Q = 10,000 个查询

```
Cache 命中率 ≈ f(Cross-Query Reuse × Single-Page Ratio × Cache Coverage)
```

| 条件 | SIFT1M | SIFT10M |
|------|--------|---------|
| Single-page postings | 30.8% | 25.6% |
| 平均访问次数 | 6.5 | 1.28 |
| Cache 容量 (256MB) | 可覆盖 hot set | 不能覆盖 |
| **实际 Cache 命中率** | **77.72%** | **18.26%** |

## 8. SIFT1M vs SIFT10M 对比总结

### 8.1 规模差异 vs 性能差异

| 维度 | 规模比 | 性能比 |
|------|--------|--------|
| 向量数 | 10x | - |
| Posting 总数 | 10x | - |
| 被访问 posting | 5x | - |
| **Cross-Query Reuse** | - | **1/5** |
| **Cache 命中率** | - | **1/4** |

### 8.2 关键差异

| 因素 | SIFT1M | SIFT10M | 对 Cache 的影响 |
|------|--------|---------|-----------------|
| Single-page postings 数量 | ~30K | 127K | 4x 更多 |
| 每个 posting 平均访问 | 6.5 次 | 1.28 次 | **5x 更少** |
| 77% posting 只访问 1 次 | 否 | **是** | 决定性 |
| Cache 能覆盖 hot set | 能 | **不能** | 失效 |

### 8.3 为什么 Top 10% I/O Wait 贡献相似？

SIFT1M 和 SIFT10M 的 Top 10% posting 贡献都是 ~28-30%：
- 两个数据集的 I/O wait 分布都较均匀 (Gini ~0.3)
- 没有集中的"坏 posting"
- 这导致 M2-H selective hybrid 在两个数据集上都失败

## 9. 论文分析要点

### 9.1 核心发现

1. **I/O Wait 分布均匀** (Gini ~0.3)
   - 没有集中的热点 posting
   - M2-H selective hybrid 不适用

2. **Cross-Query Reuse 规模依赖**
   - R = (Q × K) / N
   - SIFT10M 的 reuse 是 SIFT1M 的 1/5

3. **77% posting 只被访问 1 次**
   - 这是 cache 效率的致命因素
   - 无论 cache 多大都无法有效复用

4. **M1 Page Cache 的规模敏感性**
   - 仅在 cross-query reuse 高时有效
   - 大规模数据集失效

### 9.2 瓶颈层级图

```
┌─────────────────────────────────────────────────────────────────┐
│                     SPANN 查询瓶颈分析                           │
├─────────────────────────────────────────────────────────────────┤
│ Level 1: Batch Read / I/O         ~68%    [主要瓶颈]            │
│         ├─ Posting/page 粒度读放大                              │
│         ├─ 99.6% 扫描元素被丢弃                                  │
│         └─ Cross-query reuse 不足 (大规模)                      │
├─────────────────────────────────────────────────────────────────┤
│ Level 2: Head Search              ~26%                          │
│         ├─ Head graph 遍历                                      │
│         └─ Routing 质量                                          │
├─────────────────────────────────────────────────────────────────┤
│ Level 3: Distance Calculation     ~9%                           │
│         └─ 去重后 ~2600 次距离计算                               │
├─────────────────────────────────────────────────────────────────┤
│ Level 4: Posting Parse            ~10%                          │
│         └─ VID + payload 解析                                   │
└─────────────────────────────────────────────────────────────────┘
```

### 9.3 优化空间评估

| 方向 | 原理 | SIFT1M | SIFT10M | 规模依赖 | 可行性 |
|------|------|--------|---------|----------|--------|
| M1 Page Cache | cross-query reuse | +5% | 0% | **强** | 小规模 |
| M2-H Selective | 热点 posting | 失败 | 失败 | 无 | 不推荐 |
| M4 Dedupe | query duplicate | 失败 | 失败 | 无 | 不推荐 |

### 9.4 关键公式

**Cross-Query Reuse**:
```
R = (Q × K) / N
```
- Q = 查询数
- K = 每查询访问的 posting 数
- N = 被访问的 posting 总数

**Cache 效率预测**:
```
H ≈ min(R × S × C, 1)
```
- R = cross-query reuse
- S = single-page posting 比例
- C = cache 覆盖率

**I/O Wait 集中度**:
```
Gini 系数 < 0.4 → 分布均匀 → selective 优化无效
```
