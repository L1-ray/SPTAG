# SPANN I/O 模式分析报告

**日期**: 2026-05-03
**数据集**: SIFT10M (UInt8 + DEFAULT)
**查询数**: 10,000
**目的**: 通过实测数据分析测试时的 I/O 模式

---

## 结论

**测试是 4-16KB 小块 posting 粒度随机读为主，平均请求约 8KB**

| 指标 | 实测值 | 随机读特征 | 顺序读特征 |
|------|--------|------------|------------|
| 平均请求大小 | 7.87 KB | ✓ 小块随机读 | ✗ 通常为更大块连续读 |
| 合并率 | **0.01%** | ✓ < 5% | ✗ 应 > 20% |
| IOPS | 137,932 | ✓ 高 IOPS | ✗ 应较低 |
| 吞吐量 | 1,060 MB/s | ✓ NVMe 随机读 | ✗ 应 > 2,000 MB/s |

---

## 实验方法

通过监控 `/proc/diskstats` 在测试前后的变化，结合 trace 数据进行分析。

注意：`/proc/diskstats` 是 block layer 设备级统计，能反映设备请求数、合并数、sector 数和累计读耗时；其中 `read_ms / reads` 只能作为设备层平均服务时间的粗略近似，不能直接等同于应用层单请求 latency 或 query latency。

### 测试环境

- **磁盘**: NVMe SSD (nvme0n1p8)
- **数据文件**: `/mnt/data` 分区
- **索引大小**: 8.46 GB

### 测试条件

- SearchThreadNum = 8
- Query 数量 = 10,000
- Page Cache = OFF

## 分析

### 证据 1：平均请求大小 ≈ 8KB

| I/O 模式 | 典型请求大小 | 实测值 |
|----------|--------------|--------|
| 4KB 随机读 | ~4KB | 仅占 25.3% |
| 8KB 随机读 | ~8KB | 平均值 7.87 KB |
| 顺序读 | 通常为更大块连续读 | ✗ |
| **Posting 粒度随机读** | 4-16KB | **✓** |

**解释**：每个 posting 平均 1.95 页，请求大小 4-16KB，符合实测分布。

### 证据 2：合并率 ≈ 0%

```
合并率 = merges / reads = 35 / 639,963 = 0.01%
```

| I/O 模式 | 典型合并率 | 实测值 |
|----------|------------|--------|
| 随机读 | < 5% | **0.01% ✓** |
| 顺序读 | > 20% | ✗ |

**解释**：合并率接近零说明请求在物理位置上高度分散，无法被 I/O 调度器合并。

### 证据 3：IOPS 和吞吐量

| 指标 | 实测值 | NVMe 随机读典型特征 | NVMe 顺序读典型特征 |
|------|--------|-------------|-------------|
| IOPS | 137,932 | 100K-500K ✓ | < 50K |
| 吞吐量 | 1,060 MB/s | 500-1500 MB/s ✓ | > 2000 MB/s |

**解释**：高 IOPS、中等吞吐量符合本机 NVMe 小块随机读特征。这里的阈值是经验对比，不是严格硬件分类标准。

### NVMe SSD 性能基准测试结果

| 测试场景 | IOPS | 吞吐量 | 请求大小 |
|----------|------|--------|----------|
| 顺序读 1MB (单线程) | 4,893 | 2,438 MB/s | 510 KB |

| 4K 随机读 (8线程) | **141,908** | 554 MB/s | 4 KB |
| 8K 随机读 (8线程) | 118,960 | 929 MB/s | 8 KB |
| **SPANN 实测** | **137,932** | **1,060 MB/s** | **7.87 KB** |

### 关键对比

| 指标 | SPANN | 最接近的基准 | 差异 |
|------|-------|--------------|------|
| IOPS | 137,932 | 4K 随机 8线程 (141,908) | -3% |

| 请求大小 | 7.87 KB | 8K 随机读 (8 KB) | 匹配 |

**结论**：
- SPANN IOPS 与 4K 随机读基准几乎相同 (-3%)
- SPANN 吞吐量略高于 8K 随机读基准 (+14%)，可能来自异步并发、queue depth、请求大小混合或测试条件差异
- SPANN 吞吐量仅为顺序读的 **27%** (1060 vs 3938 MB/s)

## 为什么是随机读？

### 根本原因：Routing 的选择特性

SPANN 查询流程：
```
Query → Head Graph 遍历 → Top-K heads → 64 postings
```

**关键**：Head 选择由**距离**决定，不遵循 posting_id 顺序。

### 物理位置分散

- 每个 posting 的起始页号存储在索引头中
- Posting 按构建顺序分配物理空间
- 查询访问的 posting 在物理位置上随机分布

### 位置分散的辅助解释

以下推导依赖一个假设：物理位置与 posting_id 大致线性相关。它只能作为辅助解释；强证据仍是 diskstats 中几乎为零的合并率和 trace 中的 4-16KB 请求分布。

- 98% 的相邻 posting_id 差值 > 10,000
- 物理位置差异 = 差值 × 平均每 posting 页数
- 10,000 × 1.4 页 × 4KB ≈ 56 MB 距离
- 远超 I/O 调度器的合并窗口

## 对性能优化的启示

### 为什么 Page Cache 效果有限？

Page Cache 需要**跨查询的页面复用**。SIFT10M 的问题不是随机读本身必然导致 cache 无效，而是随机访问的 posting 工作集大且复用低：

1. 合并率 0.01% → 请求物理位置高度分散
2. 10,000 查询访问 ~500K 个 posting
3. 工作集远超 cache 容量 (256MB ≈ 65K 页)

### 性能瓶颈

| 因素 | 影响 |
|------|------|
| 随机访问模式 | I/O 调度器无法合并，NVMe 优势受限 |
| 小请求 (8KB) | 每次请求开销固定，吞吐量受限 |
| 高并发 (st=8) | 队列竞争增加延迟 |

### I/O 模式图示

```
查询请求:  [Q1] [Q2] [Q3] ... [Q10000]
              ↓   ↓   ↓
访问 posting: [P1, P5, P23, P89, ...]  ← 物理位置随机
              ↓   ↓   ↓   ↓

```

每个 posting 内部页面连续，但 posting 之间随机访问。
