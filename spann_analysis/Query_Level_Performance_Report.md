## 1. 延迟分布分析

### 1.1 总延迟统计

| 统计量 | SIFT1M | SIFT10M | 比值 |
|--------|--------|---------|------|
| **平均值** | 1.329 ms | 1.478 ms | 1.11x |
| 标准差 | 1.165 ms | 0.536 ms | 0.46x |
| 最小值 | 0.542 ms | 0.671 ms | 1.24x |
| 最大值 | 42.049 ms | 21.576 ms | 0.51x |
| P50 | 1.303 ms | 1.464 ms | 1.12x |
| P90 | 1.529 ms | 1.689 ms | 1.10x |
| P99 | 1.788 ms | 1.930 ms | 1.08x |
| P99.9 | 4.399 ms | 5.053 ms | 1.15x |

**关键观察**:
- SIFT10M 平均延迟仅比 SIFT1M 高 11%，远小于 10x 的规模差异
- SIFT10M 延迟方差更小 (0.46x)，说明查询分布更均匀
- SIFT1M 有更长的长尾 (最大 42ms vs 22ms)

### 1.2 延迟分解

| 组件 | SIFT1M 平均 | SIFT10M 平均 | SIFT1M 占比 | SIFT10M 占比 |
|------|-------------|--------------|-------------|--------------|
| **Head Search** | 0.331 ms | 0.380 ms | 24.9% | 25.7% |
| **Ex (Posting) Search** | 0.998 ms | 1.098 ms | 75.1% | 74.3% |
| └─ Batch Read (I/O) | 0.830 ms | 1.009 ms | 62.5% | 68.3% |
| └─ Posting Parse | 0.137 ms | 0.143 ms | 10.3% | 9.7% |
| └─ Distance Calc | 0.138 ms | 0.138 ms | 10.4% | 9.4% |

**关键观察**:
- Ex/Posting Search 占主导地位 (~75%)
- Batch Read (I/O) 是最大瓶颈 (63-68%)
- Head Search 开销稳定 (~25%)

## 2. I/O 特性分析

### 2.1 读取量统计

| 指标 | SIFT1M 平均 | SIFT10M 平均 | SIFT1M P99 | SIFT10M P99 | 比值 |
|------|-------------|--------------|------------|-------------|------|
| **读取页数** | 118.7 | 125.9 | 151 | 162 | 1.06x |

| **扫描元素** | 2,731 | 2,956 | 3,698 | 4,072 | 1.08x |

**关键观察**:
- 读取量仅增加 6-8%，远小于 10x 规模差异
- 说明 routing 质量保持稳定，每个查询访问的 posting 数量不变 (64)

### 2.2 效率指标

| 指标 | SIFT1M | SIFT10M |
|------|--------|---------|
| Bytes/Page | 4,096 | 4,096 |
| Elements/Page | 23.0 | 23.5 |
| Bytes/Element | 173.9 | 170.4 |

**关键观察**:
- 页面利用率一致 (~23 elements/4KB page)
- 每个元素约 174 bytes (VID + 128B payload)

### 2.3 I/O 放大分析

| 阶段 | SIFT1M | SIFT10M | 说明 |
|------|--------|---------|------|
| 读取元素 | 2,731 | 2,956 | 扫描的候选 |

**关键洞察**:
- ~99.6% 的扫描元素被丢弃
- 这是 posting/page 粒度读放大的直接体现

### 6.1 相关性矩阵 (SIFT10M)

## 7. SIFT1M vs SIFT10M 对比总结

### 7.1 规模差异 vs 性能差异

| 维度 | 规模比 | 性能比 |
|------|--------|--------|
| 向量数 | 10x | - |
| 索引大小 | 10x | - |
| **平均延迟** | - | **1.11x** |
| **页数/查询** | - | **1.06x** |
| **Recall** | - | **0.97x** |

**结论**: SPANN 的查询性能随规模增长**亚线性**，这是其核心优势。

### 7.2 为什么延迟增长远小于规模增长？

1. **Routing 选择性**: 每个查询只访问固定的 64 个 posting
2. **Posting 大小稳定**: 平均每 posting 元素数接近
3. **Page 对齐开销一致**: 4KB 页面粒度不变

### 7.3 为什么 M1 Page Cache 在 SIFT10M 失效？

**前提条件**: Q = 10,000 个查询

| 因素 | SIFT1M | SIFT10M | 影响 |
|------|--------|---------|------|
| Single-page postings | 48,158 | 440,342 | 9.1x 更多 |
| 每个 posting 平均访问 (Q=10,000) | 4.26 次 | 0.43 次 | **10x 更少** |
| Cache 容量 | 256 MB | 256 MB | 相同 |
| **Hot working set** | **可覆盖** | **不可覆盖** | **决定性** |

**根因**:
- Cross-query page reuse = (查询数 × 每查询访问的 posting 数) / posting 总数
- SIFT1M: 10,000 × 64 / 150,076 = **4.26 次/posting**
- SIFT10M: 10,000 × 64 / 1,496,408 = **0.43 次/posting**
- Cache 需要跨查询复用才能有效，SIFT10M 在 Q=10,000 条件下复用率太低

## 8. 论文分析要点

### 8.1 核心发现

1. **SPANN 的亚线性扩展性**: 10x 规模 → 11% 延迟增长
2. **Posting/page 粒度读放大**: ~99.6% 的扫描元素被丢弃
3. **I/O 主导**: Batch Read 占总延迟 63-68%
4. **Cache 有效性的规模依赖**: Cross-query reuse 随规模下降

### 8.2 性能瓶颈层级

```
瓶颈层级 (按影响排序):
1. Batch Read / I/O        ~68% ← 主要瓶颈
2. Head Search             ~26%
3. Distance Calculation    ~9%
4. Posting Parse           ~10%
```

### 8.3 优化空间评估

| 优化方向 | SIFT1M 有效性 | SIFT10M 有效性 | 规模依赖 |
|----------|---------------|----------------|----------|
| M1 Page Cache | ✓ +5% | ✗ +0.2% | 强 |
| M2-H Selective Hybrid | ✗ | ✗ | 无 |
| M4 Primary-Secondary | ✗ | ✗ | 无 |

**结论**: 当前结构下，小规模数据集的优化空间有限，大规模数据集的优化空间更有限。
